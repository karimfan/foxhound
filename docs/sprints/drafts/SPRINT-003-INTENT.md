# Sprint 003 Intent: Hierarchical Codebase Wiki Generator

## Seed

Build a hierarchical documentation tool that traverses the project directory tree from leaves to root. At each directory, read source files, generate a markdown summary of what the code does with links to files, and create a `SUMMARY.md` in that directory. These summaries chain upward — parent directories include summaries of their children. The top-level output is `wiki.md`. Directories `.claude`, `.codex`, and `docs` are excluded. The tool should then be run against the leash codebase.

## Context

- **Leash** is a container-based AI agent security platform written primarily in Go, with Swift (macOS), TypeScript (control UI), C (eBPF), and Python/Shell (build scripts)
- The codebase has ~20+ directories with clear module boundaries (`internal/runner`, `internal/proxy`, `internal/lsm`, etc.)
- No existing auto-documentation tooling exists in the project
- Sprint 001 established Go as the preferred language for project tooling (rewrote Python tracker to Go)
- Sprint 002 was a non-code analytical sprint

## Recent Sprint Context

- **Sprint 001**: Rewrote sprint tracker from Python to Go as a single-file CLI. Established the pattern of Go-based project tooling.
- **Sprint 002**: GTM strategy analysis (non-code sprint, appended to thesis document).
- This is the first tooling sprint since 001, and the second code sprint overall.

## Relevant Codebase Areas

- **Directory structure**: `cmd/`, `internal/` (19 packages), `build/`, `controlui/`, `e2e/`, `mac-leash/`, `npm/`, `.github/`
- **Language detection needed**: `.go`, `.swift`, `.ts`, `.tsx`, `.c`, `.py`, `.sh`, `.js`
- **Existing tooling pattern**: `docs/sprints/tracker.go` — single-file Go CLI, no external deps
- **Build system**: `Makefile` at root, `go.mod` for Go deps
- **Mixed content**: Some directories have configs (JSON, YAML, TOML) alongside source code

## Constraints

- Must follow the project pattern of single-file Go tooling (per Sprint 001 precedent)
- Must exclude `.claude`, `.codex`, and `docs` directories
- Must traverse leaf-to-root (bottom-up) so child summaries are available when generating parent summaries
- Must handle multiple languages (Go, Swift, TypeScript, C, Python, Shell)
- Must produce valid markdown with relative links between files
- Generated summaries should be useful for onboarding and navigation
- The tool itself should be runnable via `go run`

## Success Criteria

1. A Go CLI tool that can be pointed at any directory and generates hierarchical markdown docs
2. Running it against the leash codebase produces accurate, useful summaries
3. `wiki.md` at the root provides a navigable overview of the entire codebase
4. Each subdirectory gets a `SUMMARY.md` linking to its files and child summaries
5. The tool handles the mixed-language nature of the project gracefully
6. Summaries are concise but informative — not just file listings, but descriptions of what the code does

## Open Questions

1. **LLM-powered or static analysis?** Should summaries be generated by an LLM (richer descriptions) or via static analysis/heuristics (faster, no API dependency)? The seed implies understanding "what the code does" which suggests LLM involvement.
2. **Output file naming**: `SUMMARY.md` per directory, or a different name to avoid conflicts?
3. **Incremental updates**: Should the tool support re-running on changed files only, or always regenerate everything?
4. **File size limits**: How to handle large files — summarize first N lines, or the whole file?
5. **Config files**: Should non-source files (Dockerfile, Makefile, YAML configs) be included in summaries?
6. **Go tool or standalone binary?**: Single-file `go run` like tracker.go, or a proper cmd/ entry point?
